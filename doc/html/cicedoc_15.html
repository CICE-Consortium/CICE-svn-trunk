Execution procedures

To compile and execute the code, edit the grid size and other parameters in source/ice_model_size.F as desired. Set the internal parallelization method and number of processors (BINTYPE, NX, NY) in IRIX64.defs (or make an equivalent file for another architecture), and modify the INCLUDE directory path and other settings for your system. NX and NY should evenly divide the respective number of grid points in each direction. We suggest that NY<=2 for load balancing; otherwise processors assigned subdomains near the equator have little work to do (we hope).

For stand-alone runs, check that -Dcoupled is not set in the IRIX64.defs GDEFS. Then type gmake to compile (gmake clean will remove much of what's been created to this point). Alter directory paths, file names and the execution command as desired in run_ice, and finally type run_ice to execute. (There may be an additional operating system command. For example, on the LANL SGIs one types bsub < run_ice to submit the job to the batch queue.)

For coupled runs, check that -Doceanmixed is not set in the IRIX64.defs GDEFS, and that -Dcoupled is. You may compile the model as above or use NCAR scripts that set up and compile all of the CCSM coupled model components at once (not available in this distribution).

This procedure creates the output log file ice.log.[timestamp], and if npt is long enough compared with dumpfreq and histfreq, dump files iced.[timeID] and netCDF history output files iceh_[timeID].nc. Using the <4/3o> grid, the log file should be similar to ice.log.mpi_4x2, provided for the user's convenience. ice.log.mpi_4x2 is the log file from an MPI run on 8 processors (NX=4 and NY=2), on the <4/3o> grid.

CICE namelist variables available for changes after compile time appear in ice.log.* with values read from the file ice_in; their definitions are given in the index. For example, to run for a different length of time, say three days, set npt=36 in ice_in. At present, the user supplies the time step dt and the number of subcycles ndte, and dte is then calculated in subroutine init_evp. The primary reason for doing it this way is to ensure that ndte is an integer.

To restart from a previous run, set the filename in ice.restart_file (created by the previous run) to the desired data file (iced.[timeID]), then set restart=.true. in ice_in. Restarts are exact for MPI or single processor runs.

The structure and flow of the sea ice code, described below, is fairly well outlined in the main driver routine ice.F. Note that the thermodynamics routine is broken into two pieces, so that fluxes can be returned to the coupler as quickly as possible. This enables the flux coupler to deliver the thermodynamic fluxes to other component models while the ice model continues running. (Subroutines listed here may be called from numerous other places during a run.)
